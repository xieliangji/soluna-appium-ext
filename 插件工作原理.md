# soluna-ext 与 Appium 的协作原理

本文解释这个插件在 Appium 启动与运行时是如何接入、如何处理请求、以及关键设计点。

## 1. 插件如何被 Appium 识别并加载

在 `package.json` 中声明了：

- `appium.pluginName`: `soluna-ext`
- `appium.mainClass`: `SolunaExtPlugin`

Appium 在启动并启用该插件时，会加载 `SolunaExtPlugin` 类（`lib/plugin.ts`），并调用其静态方法 `updateServer(...)`。

## 2. updateServer 是协作入口

`SolunaExtPlugin.updateServer(expressApp, httpServer)` 是插件与 Appium Server 集成的核心入口。这里做了两件事：

1. 运行启动前检查（preflight）  
   `runPreflightChecks()` 确保宿主机依赖可用：
   - `adb`
   - `go-ios` 或 `ios`

2. 向 Appium 使用的 Express 应用挂载自定义路由  
   - `GET /soluna/device`
   - `GET /soluna/devices`
   - `POST /soluna/command`

这意味着插件并不是独立 HTTP 服务，而是“扩展”了 Appium 的同一个 HTTP 服务。

## 3. 启动前检查如何阻断 Appium 启动

preflight 在 `lib/cli/preflight.ts`：

- 命令存在性检查按平台选择：
  - Windows: `where`
  - macOS/Linux: `which`
- 如果缺少必要命令，会 `console.error` 并 `throw Error`

因为这个异常发生在 `updateServer` 阶段，Appium 会把它视为插件初始化失败，从而阻止服务以“可用状态”启动。

## 4. 设备查询请求的协作链路

### 4.1 `GET /soluna/device?udid=...`

处理路径：

1. `handleGetDeviceInfo`（路由层）校验 `udid`
2. 调用 `lookupDeviceByUdid`（服务聚合层）
3. 先查 Android，再查 iOS：
   - Android：`adb devices` + `adb shell getprop`
   - iOS：`go-ios|ios list --details`
4. 统一映射为 `UnifiedDeviceInfo`
5. 返回 Appium 风格响应：
   - 200: `{value: {exists: true, device: ...}}`
   - 404: `{value: {exists: false, message: ...}}`
   - 400: `{value: {error: "invalid_argument", ...}}`

### 4.2 `GET /soluna/devices`

处理路径：

1. `handleListDevices`
2. `listConnectedDevices` 并发聚合：
   - `listAndroidDevices`
   - `listIosDevices`
3. 返回：
   `{value: {count, devices}}`

## 5. 命令执行请求的协作链路

`POST /soluna/command` 的链路：

1. `handleExecuteCommand` 校验 body（tool/args/timeout/maxOutput）
2. `executeSupportedCommand` 执行命令
3. 内部通过 `spawn` 运行子进程，支持：
   - 超时控制（`SIGTERM` -> `SIGKILL`）
   - 输出长度上限截断
   - go-ios/ios 命令回退
4. 返回：
   - `exitCode === 0` -> HTTP 200
   - 非 0 -> HTTP 422（仍带完整执行结果）
   - 执行异常 -> HTTP 500 + `command_execution_failed`

## 6. 与 Appium 日志系统的协作

命令路由中增加了日志埋点（`lib/http/command-route.ts`）：

- `info`：请求摘要与执行摘要
- `debug`：`stdout/stderr` 全量输出
- `warn/error`：参数校验失败和执行失败

这些日志进入 Appium 的统一日志流，便于和 Appium 主流程日志一起排查问题。

## 7. 为什么要分层（route / service / cli）

当前代码把职责分成三层：

- `http/*`：只处理 HTTP 协议与响应结构
- `services/*`：处理业务编排与平台差异
- `cli/*`：处理外部命令执行能力

好处是：

1. 更容易测试（可以注入 `CommandRunner`）
2. 更容易扩展（新增平台或新接口时影响面小）
3. 更符合 Appium 插件“扩展能力，而非改写核心”的模式

## 8. 一句话总结

这个插件通过 `updateServer` 把自身挂到 Appium 的 Express 服务器里，在启动时做环境守卫（preflight），在运行时提供设备查询与命令执行能力，并把输出统一到 Appium 的日志与响应风格中。
