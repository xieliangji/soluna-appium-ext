# soluna-ext 与 Appium 协作时序图

本文给出“启动阶段 + 3 个接口”的时序图版本，便于快速理解调用链。

## 1) Appium 启动与插件挂载（含 preflight）

```mermaid
sequenceDiagram
    autonumber
    participant CLI as appium CLI
    participant Appium as Appium Server Core
    participant Plugin as SolunaExtPlugin.updateServer
    participant Preflight as runPreflightChecks
    participant HostCLI as Host CLI (adb / go-ios / ios)
    participant Express as Appium Express App

    CLI->>Appium: appium --use-plugins=soluna-ext
    Appium->>Plugin: invoke updateServer(expressApp, httpServer)
    Plugin->>Preflight: runPreflightChecks()

    Preflight->>HostCLI: which/where adb
    Preflight->>HostCLI: which/where go-ios (fallback ios)

    alt missing required CLI
        Preflight-->>Plugin: throw Error
        Plugin-->>Appium: plugin init failed
        Appium-->>CLI: startup blocked
    else all required CLI present
        Preflight-->>Plugin: OK
        Plugin->>Express: register GET /soluna/device
        Plugin->>Express: register GET /soluna/devices
        Plugin->>Express: register POST /soluna/command
        Plugin-->>Appium: updateServer done
        Appium-->>CLI: server ready
    end
```

## 2) `GET /soluna/device?udid=...`

```mermaid
sequenceDiagram
    autonumber
    participant Client as HTTP Client
    participant Appium as Appium Express Router
    participant Route as handleGetDeviceInfo
    participant Service as lookupDeviceByUdid
    participant Android as android service
    participant IOS as ios service
    participant HostCLI as adb / go-ios|ios

    Client->>Appium: GET /soluna/device?udid=...
    Appium->>Route: handle request

    alt udid missing
        Route-->>Client: 400 {value:{error:"invalid_argument"}}
    else udid exists
        Route->>Service: lookupDeviceByUdid(udid)
        Service->>Android: findAndroidDeviceByUdid
        Android->>HostCLI: adb devices + adb -s <udid> shell getprop

        alt found on Android
            Android-->>Service: UnifiedDeviceInfo(android)
            Service-->>Route: found
            Route-->>Client: 200 {value:{exists:true,device}}
        else not found on Android
            Service->>IOS: findIosDeviceByUdid
            IOS->>HostCLI: go-ios|ios list --details

            alt found on iOS
                IOS-->>Service: UnifiedDeviceInfo(ios)
                Service-->>Route: found
                Route-->>Client: 200 {value:{exists:true,device}}
            else not found
                Service-->>Route: found=false
                Route-->>Client: 404 {value:{exists:false,message}}
            end
        end
    end
```

## 3) `GET /soluna/devices`

```mermaid
sequenceDiagram
    autonumber
    participant Client as HTTP Client
    participant Appium as Appium Express Router
    participant Route as handleListDevices
    participant Service as listConnectedDevices
    participant Android as listAndroidDevices
    participant IOS as listIosDevices
    participant HostCLI as adb / go-ios|ios

    Client->>Appium: GET /soluna/devices
    Appium->>Route: handle request
    Route->>Service: listConnectedDevices()

    par Android branch
        Service->>Android: listAndroidDevices()
        Android->>HostCLI: adb devices
        Android->>HostCLI: adb -s <serial> shell getprop (per device)
        Android-->>Service: UnifiedDeviceInfo[]
    and iOS branch
        Service->>IOS: listIosDevices()
        IOS->>HostCLI: go-ios|ios list --details
        IOS-->>Service: UnifiedDeviceInfo[]
    end

    Service-->>Route: merge android + ios
    Route-->>Client: 200 {value:{count,devices}}
```

## 4) `POST /soluna/command`

```mermaid
sequenceDiagram
    autonumber
    participant Client as HTTP Client
    participant Appium as Appium Express Router
    participant Route as handleExecuteCommand
    participant Executor as executeSupportedCommand
    participant Proc as child_process.spawn
    participant HostCLI as adb / go-ios / ios
    participant Log as Appium Log Stream

    Client->>Appium: POST /soluna/command
    Appium->>Route: handle request
    Route->>Route: validateCommandRequest(body)

    alt invalid body
        Route->>Log: warn validation failed
        Route-->>Client: 400 {value:{error:"invalid_argument"}}
    else valid body
        Route->>Log: info request summary
        Route->>Executor: executeSupportedCommand(...)
        Executor->>Proc: spawn candidate command
        Proc->>HostCLI: run command with args
        Proc-->>Executor: stdout/stderr/exitCode (+ timeout/truncate)
        Executor-->>Route: CommandExecuteResult
        Route->>Log: info result summary
        Route->>Log: debug stdout/stderr

        alt exitCode == 0
            Route-->>Client: 200 {value:{ok:true,...result}}
        else exitCode != 0
            Route-->>Client: 422 {value:{ok:false,...result}}
        end
    end

    opt unexpected execution exception
        Route->>Log: error execution failed
        Route-->>Client: 500 {value:{error:"command_execution_failed"}}
    end
```

## 5) 阅读建议

- 先看“启动时序图”，理解插件何时生效、何时阻断启动。
- 再看 3 个请求图，分别对应查询单设备、查询全设备、执行命令。
- 如果排查问题，建议把“路由层返回状态码 + Appium 日志级别”结合来看。
